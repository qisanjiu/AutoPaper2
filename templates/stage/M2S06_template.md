## 2. 执行顺序与分支逻辑

```
Phase 1: Exp-1 (主实验)
  ├── 如果方法显著优于所有 baseline → 进入 Phase 2
  ├── 如果方法略优但不显著 → 诊断：超参/实现/设计？
  │   ├── 如果是超参问题 → FIX 超参 → 重跑 Exp-1
  │   ├── 如果是实现 bug → BACKTRACK → M3S01 (代码生成)
  │   └── 如果是设计问题 → BACKTRACK → M2S03 (重新设计)
  └── 如果方法不优于 baseline → 诊断根因
      ├── 根因在实现 → BACKTRACK → M3S01
      ├── 根因在设计 → BACKTRACK → M2S03
      └── 根因在假设 → BACKTRACK → M1S04

Phase 2: Exp-2,3 (消融实验)
  ├── 如果消融结果与预期一致 → 进入 Phase 3
  └── 如果消融结果异常 → 诊断组件实现 → BACKTRACK → M3S01

Phase 3: Exp-4 (鲁棒性/进一步分析)
  └── 完成 → 进入 M3 结果验证
```

## 3. 成功/失败判定标准

### 3.1 主实验成功标准
- [ ] 方法在主要指标上**显著优于**最强 baseline（统计检验 p < 0.05）
- [ ] 提升幅度**有实际意义**（effect size > 0.5）
- [ ] 固定 seed=42 的主结果已完成并可复现命令

### 3.2 失败判定标准（触发 BACKTRACK）
| 失败类型 | 判定条件 | 回溯目标 |
|---------|---------|---------|
| 假设否定 | 方法不优于 baseline，且不是实现问题 | M1S04 |
| 设计缺陷 | 消融显示核心组件无贡献 | M2S03 |
| 实现 bug | 代码逻辑错误导致性能下降 | M3S01 |
| 超参问题 | 方法潜在有效但超参未调好 | M2S05 |
| 数据问题 | 数据集与任务不匹配 | M2S02 |

## 4. 风险与应对

| 风险 | 概率 | 影响 | 应对措施 | 触发条件 |
|------|------|------|---------|---------|
| Baseline 代码不可运行 | 中 | 高 | 自行实现或更换 baseline | M3S01 发现 |
| 训练时间过长 | 中 | 中 | 减少 epoch 或使用小规模数据集预实验 | Exp-1 预实验 |
| 显存不足 | 中 | 高 | 减小 batch size 或模型规模 | 运行时报错 |
| 方法效果不达预期 | 中 | 高 | 按§3.2回溯策略处理 | Exp-1 结果 |
| 固定 seed 配置错误 | 低 | 中 | 统一设为 seed=42 并记录命令 | 运行日志与配置检查 |
