## 二、Stage 详细设计

### M2S01: Problem-Driven Cross-Domain Literature Search
**问题驱动的跨领域文献搜索**

#### 目标
基于 M1 识别的核心 Gap，进行**跨领域/弱相关文献搜索**，寻找潜在的解决思路和方法。

#### 输入
- `knowledge/handoff_M1_M2.md` → 核心 Gap 列表（按优先级排序）
- `knowledge/M1/M1S02_literature_deepdive.md` → §4 Gap 分析（了解"已有什么"、"缺什么"）
- `knowledge/M1/M1S03_research_question.md` → 研究问题定义
- `knowledge/M1/M1S04_hypothesis_generation.md` → 核心假设

#### 核心任务

**Step 1: Gap → 技术问题拆解**
```
Gap-1: "现有方法缺乏自适应调制能力"
  └─► 技术问题: "如何根据信道状态动态调整传输策略？"
  └─► 子问题: "动态性如何建模？" "调制策略如何学习/优化？"

Gap-2: "现有方法在高噪声下性能急剧下降"
  └─► 技术问题: "如何提升方法在恶劣信道条件下的鲁棒性？"
  └─► 子问题: "噪声建模？" "鲁棒性如何量化和优化？"
```

**Step 2: 跨领域搜索策略**

对每个技术问题，按以下维度搜索弱相关领域：

| 搜索维度 | 说明 | 示例（图像语义通信） |
|---------|------|---------------------|
| **同模态不同任务** | 相同数据模态，不同任务目标 | 计算机视觉中的自适应编码 → 语义通信自适应调制 |
| **同任务不同模态** | 相同任务目标，不同数据模态 | NLP 中的语义压缩 → 图像语义压缩 |
| **底层原理相通** | 数学/信息论/优化原理相通 | 信息论中的率失真优化 → 通信资源分配 |
| **相似结构问题** | 问题结构相似（动态、鲁棒、稀疏等） | 强化学习中的自适应策略 → 通信自适应调制 |

**Step 3: 候选方案收集与初筛**

```markdown
## 候选方案池

### 方案 1: [来源领域] → [目标问题]
- **来源论文**: [标题, 作者, 年份, Venue]
- **原问题**: 该论文解决什么问题？
- **核心思想**: 1-2 句话概括其核心方法
- **与本问题关联**: 为什么这个思想可能解决我们的 Gap？
- **初步可行性**: 高/中/低
- **需要适配度**: 高（需大量修改）/ 中（部分修改）/ 低（可直接应用）

### 方案 2: ...
```

...

- [ ] 候选方案池 ≥ 3 个（含领域内改进方案）

- [ ] 候选方案按可行性排序，明确重点分析对象

#### 可回溯原因
- 找不到任何跨领域解决思路 → BACKTRACK → 扩大搜索范围 或 BACKTRACK → M1S02（补充领域知识）
- 所有候选方案适配难度过高 → BACKTRACK → M1S03（调整问题范围）

### M2S02: Method Inspiration & Adaptation Analysis
**方法启发与适配分析**

#### 目标
对 M2S01 筛选出的 **1-3 个重点候选方案** 进行深入分析，明确：**该工作的思想如何迁移到本问题？需要哪些改进？**

#### 输入
- `knowledge/M2/M2S01_cross_domain_search.md` → 候选方案池 + 排序
- `knowledge/M1/M1S02_literature_deepdive.md` → 了解本领域技术细节（便于评估适配可行性）

#### 核心任务

**Step 1: 单篇深入分析**

对每篇重点论文，完成以下分析：

```markdown
## 论文深入分析: [标题]

### 1. 论文核心贡献
- 原问题: ...
- 核心方法: ...
- 关键创新点: ...

### 2. 思想迁移分析
#### 2.1 问题结构映射
```
原论文问题结构          本问题结构
├─ 输入: ...     ───►   ├─ 输入: ...
├─ 输出: ...     ───►   ├─ 输出: ...
├─ 约束: ...     ───►   ├─ 约束: ...
└─ 目标: ...     ───►   └─ 目标: ...
```
结构相似度: 高/中/低

#### 2.2 核心机制映射
- 原论文机制: ...
- 迁移到本问题后的机制: ...
- 映射合理性论证: ...

#### 2.3 需要适配的关键差异
| 差异维度 | 原论文 | 本问题 | 适配策略 |
|---------|--------|--------|---------|
| 数据特性 | ... | ... | ... |

- 核心组件: ...
- 算法流程（初步）: ...
- 预期优势: ...
- 潜在风险: ...
```

**Step 2: 多方案对比与选择**

```markdown
## 方案对比

| 维度 | 方案 A (论文X) | 方案 B (论文Y) | 方案 C (领域内改进) |
|------|---------------|---------------|-------------------|
| 解决 Gap | Gap-1, Gap-2 | Gap-1 | Gap-2 |
| 创新性 | 高（跨领域首次应用） | 中 | 低 |
| 适配难度 | 中 | 高 | 低 |
| 理论保证 | 有 | 弱 | 有 |
| 实现复杂度 | 中 | 高 | 低 |

**选择**: 方案 A 为主，方案 B 的某组件为辅
**理由**: ...
```

**Step 3: 方法启发总结**

明确回答：
1. **本文方法的核心思想来源是什么？**（具体到某篇论文的某个机制）

3. **为什么这个改进是必要的？**（原方法直接应用有什么问题？）
4. **改进后的方法贡献是什么？**（新在哪里？）

## 三、Gate G2 设计

### 审查维度

| Critic | 审查重点 | 通过标准 |
|--------|---------|---------|
| **Logic** | Gap→搜索→映射→设计的逻辑链条是否完整自洽 | 每个 Gap 都有对应的解决路径 |
| **Method** | 方法是否严谨、形式化描述是否完整、与现有工作对比是否诚实 | 方法可复现、对比具体 |
| **Novelty** | 跨领域迁移是否有实质性创新、是否为简单套用 | 有明确改进点、非直接移植 |

### Novelty Critic 特殊审查项（针对跨领域迁移）

```markdown
## Novelty Assessment for Cross-Domain Method

### 1. 思想来源追溯
- [ ] 核心思想来源论文已明确标注
- [ ] 来源论文的原问题与本问题的差异已说明

### 2. 改进充分性评估
- [ ] 原方法直接应用到本问题会存在什么问题？
- [ ] 我们的改进是否真正解决了这些问题？
- [ ] 改进是"必要的适配"还是"装饰性的修改"？

### 3. 创新性判定
- [ ] 如果是"首次将 A 应用于 B"：为什么之前没人做？是真的有价值还是只是没人在意？
- [ ] 如果是"改进 A 以适配 B"：改进的技术深度如何？是参数调优还是机制创新？

### 4. 诚实性检查

## 五、Skill 调用规范

### Method Agent 在不同 Stage 的职责差异

| Stage | 核心能力 | 工具需求 |
|-------|---------|---------|
| M2S01 | 跨领域文献搜索、问题拆解 | ReadFile, WriteFile, **WebSearch**, **WebFetch** |
| M2S02 | 单篇深入分析、思想映射、方案对比 | ReadFile, WriteFile, WebFetch |
| M2S03 | 形式化描述、伪代码设计、理论分析 | ReadFile, WriteFile, Shell |
| M2S04 | 数据集评估、baseline 选择、协议设计 | ReadFile, WriteFile, WebSearch |
| M2S05 | 计划整合、风险评估 | ReadFile, WriteFile |

### Subagent 协作点

```
M2S01 完成后
  └──► 创建 "Search Review Subagent"
       → 审查：跨领域搜索是否充分？是否有遗漏的重要领域？
       → 产出：搜索质量评估报告

M2S02 完成后
  └──► 创建 "Adaptation Review Subagent"
       → 审查：思想映射是否合理？改进是否充分？
       → 产出：适配合理性评估

M2S03 完成后
  └──► 创建 "Method Validation Subagent"
       → 审查：形式化描述是否完整？伪代码是否可复现？
       → 产出：方法验证报告

## 六、回溯策略

### M2 内部回溯

```
M2S05 发现实验计划不可行
  ├── 实验资源不足 → 回溯到 M2S05 调整计划
  ├── baseline 无法实现 → 回溯到 M2S04 更换 baseline
  ├── 方法过于复杂难以实现 → 回溯到 M2S03 简化方法
  ├── 方法设计发现根本缺陷 → 回溯到 M2S02 重新选择方案
  └── 所有方案都不合适 → 回溯到 M2S01 扩大搜索范围
```

### 跨模块回溯

```
Gate G2 或 M2S02 发现根本性问题
  ├── 核心假设与所有搜索到的方案冲突 → BACKTRACK → M1S04
  ├── 问题定义导致无法找到合适方案 → BACKTRACK → M1S03
  └── 领域知识不足导致搜索失败 → BACKTRACK → M1S02
```
